Agentic Project Template · Team Walkthrough

Building enterprise-grade software with agents

A reference scaffold: the files, gates, and context that keep AI-written code consistent and enterprise-grade, on a new project or a brownfield one you want to improve.

The idea. Agents write a lot of code, and they write it fast. Without guardrails, that speed produces inconsistency. The fix is not to slow the agent down. It is to make the standards executable and the context explicit, so the correct way to write code is also the only way that passes.

Q1
Aligned and enforced
How do I get enterprise-grade pull requests from agents?
Two moves: give the agent the team's standards before it writes, then make those standards impossible to skip after it writes.

1Make the context explicit

The repo holds its own knowledge: the team's best thinking, the layout of the project, and the things not to do. There is one place to update when the way you work changes.

2Enforce with gates, not good intentions

Done=pnpm verifyGREEN+pnpm e2eGREEN
No task is done until both pass, whoever or whatever wrote the code.
  • Fail fast. Every standard is an executable gate, run at the earliest point it can fail.
  • No skipping. A PreToolUse hook refuses --no-verify.
  • No "works on my machine." Local hooks and CI run the exact same checks.

Quality Gates

Standards fail early and locally, then enforce trust at the end. The same checks run locally and in CI; red means fix the cause, never weaken the config.
The catch: your context files and your codebase have to speak the same language. If the foundations hold three different patterns while AGENTS.md defines one, the agent struggles to remember which it is meant to follow, and the mess wins. So before the gates and the context can do their job, the foundations have to be fixed. That starts with a health check.
Q2
Foundation up
How do I check the health of an enterprise codebase?
Health has an order, and it does not start with the agentic setup. It starts with the behaviour map and end-to-end tests, because nothing else is safe to assess or change without them. Then the architecture, then maintainability and readability, then security, then the non-functionals. You work each layer in that order, because a weak floor makes everything above it shaky. The agentic infrastructure is not the headline here. It is what locks each fix in so it cannot regress.

The order of a health check

Audit and fix from the foundation up. The agentic infrastructure is the base plate that keeps each layer from sliding back.
The catch: a health check tells you where the floor is weak, but knowing is not fixing. Shifting a legacy codebase up to that standard means changing real code without breaking the product, and that has an order of its own. That is the next question.
Q3
In this order
How do I move a legacy brownfield codebase to modern enterprise-grade?
You do not start by refactoring. Before touching a line of code, you map the behaviours with tests, stand up the quality gates and visual regression, define the ideal architecture and conventions in .agents/, and set the definition of done. Once you are confident you will not break the user experience, you refactor toward the ideal. Then you tighten, then you prune dependencies, then you optimise. Each step unlocks the next.

The order of the shift

Foundations first, optimisation last. You only earn the right to the later steps by doing the earlier ones.

Two Guards, One Free Interior to Reshape

Behaviour and style regression are peers. Both hold the product steady from the outside, so the inside is safe to change.

The Agentic Audit Loop

With the guards in place, the refactor runs as a loop. Audit against your intended architecture, fix the gap, re-audit, and measure the progress each pass.

The architecture audit reads your intention files in .agents/ as its north star, and falls back to opinionated defaults where you have not set one. So you refactor toward the standard you defined, and each pass closes the gap. The reports were not always green: the first run found missing static analysis, a live tmp advisory, and weaker selectors, and the committed history shows each one closed.
